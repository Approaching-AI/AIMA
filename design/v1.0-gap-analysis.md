# AIMA v1.0 Gap Analysis Report

> Based on PRD v1.0 deliverables vs current codebase (develop branch, 2026-03-05)
> Current version: v0.0.x (pre-release, 61 MCP tools operational)

---

## 1. Executive Summary

### Completion Metrics

| Status | Count | Percentage |
|--------|-------|------------|
| Complete | 3 | 18% |
| Partial | 8 | 47% |
| Missing | 6 | 35% |
| **Total v1.0 deliverables** | **17** | — |

### Key Findings

1. **L0-L2 foundation is solid.** Hardware detection (8 GPU vendors), multi-runtime (K3S/Docker/Native),
   knowledge-driven config resolution, and 61 MCP tools are production-grade.
2. **L3 Agent is reactive only.** The Go Agent (L3a) has a working tool-calling loop but zero proactive
   capabilities — no patrol, no auto-tuning, no self-healing. ZeroClaw (L3b) is scaffolded but non-functional.
3. **Knowledge query engine is complete.** All 6 query types (Search/Compare/Gaps/Similar/Lineage/Aggregate)
   are implemented with SQL-backed operations.
4. **Critical gap: YAML fields defined but unused.** Several performance/constraint fields
   (`startup_time_s`, `cold_start_s`, `power_modes`, `compute_units`) are loaded into SQLite but
   never used by the resolver or deployment pipeline for constraint-based decisions.
5. **No central knowledge server.** Offline export/import works, but push/pull sync and the
   PostgreSQL aggregation service are entirely absent.
6. **No tunnel or remote access.** Fleet works over LAN (mDNS), but no WAN connectivity.

## 2. Layered Design Review

PRD defines a 5-layer intelligence stack (L0-L3b). Understanding which gaps belong to which layer
clarifies implementation approach:

```
Layer    │ Scope                           │ Implementation   │ Gap Status
─────────┼─────────────────────────────────┼──────────────────┼───────────
L0       │ Hardcoded defaults (YAML)       │ Go code          │ COMPLETE
L1       │ Human CLI overrides             │ Go code          │ COMPLETE
L2a      │ Engine Asset matching           │ Go code          │ COMPLETE
L2b      │ Model Asset variant selection   │ Go code          │ COMPLETE
L2c      │ Tuning optimal (from benchmark) │ Go code + Agent  │ PARTIAL — DB exists, no auto-promote
L3a      │ Go Agent (tool-calling loop)    │ Go code          │ PARTIAL — reactive only
L3b      │ ZeroClaw sidecar               │ External binary  │ SCAFFOLDED — lifecycle mgmt only
```

**Gap distribution by layer:**
- L0-L2b gaps (Go deterministic logic): S4, S5, S6, D3 amplifier eval, D5, K4, F4, F5, I5, I6
- L2c-L3 gaps (Agent dynamic optimization): A2, A3, A4, A5, K5 auto-promote, K6
- Infrastructure gaps (new services): K9, Tunnel

## 3. Detailed Gap Analysis by Domain

### 3.1 Supply — Resource Awareness

#### S1: Auto-detect GPU/NPU/CPU, generate capability vector — MVP, P0
- **Status: COMPLETE**
- `internal/hal/hal.go`: `DetectHardware()` returns full `HardwareInfo` struct
- `internal/hal/gpu.go`: Table-driven GPU enrichers (`gpuEnrichers` map) support 8 vendors:
  NVIDIA (`nvidia-smi`), AMD ROCm (`rocm-smi`), Hygon DCU, Apple Metal, Ascend (`npu-smi`),
  Moore Threads (`mtml`), Iluvatar (`ixsmi`), Enflame (`efml`)
- 8 Hardware Profile YAML files in `catalog/hardware/`

#### S2: Real-time resource monitoring (VRAM/CPU/RAM) — MVP, P0
- **Status: COMPLETE**
- `internal/hal/hal.go`: `CollectMetrics()` gathers GPU memory, power draw, temperature
- MCP tool: `device.metrics` exposes real-time data
- Web UI dashboard displays live metrics

#### S3: Detect device TDP / power mode — MVP, P1
- **Status: PARTIAL**
- HAL collects `PowerDrawWatts` for all GPU vendors (`internal/hal/gpu.go:190,238,448,588,695,758,819,865`)
- Hardware Profile YAML has `constraints.tdp_watts` and `constraints.power_modes`
- **Missing:** Power mode is recorded but never used for constraint-based decisions in the resolver

#### S4: Resource budget estimation (pre-deploy prediction) — v1.0, P1
- **Status: MISSING**
- `CheckFit()` (`internal/knowledge/resolver.go:604`) does post-resolve GPU memory adjustment,
  but there is no pre-deployment resource cost prediction model
- No `cost(path, R)` function as described in PRD §3

#### S5: Abstract ResourceSlot with pluggable backends — v1.0, P1
- **Status: MISSING**
- `PartitionSlot` (`internal/knowledge/resolver.go:31`) is a flat struct with GPU/CPU/RAM fields
- No `ResourceSlot` interface; no pluggable backend abstraction (MIG/MPS/HAMi)
- Runtime partition is passed as `PartitionRequest` (`internal/runtime/runtime.go:69`) directly

#### S6: Engine-expanded resource boundary modeling (effective_R) — v1.0, P2
- **Status: MISSING**
- No `effective_R(engine, R)` computation
- CPU offload capability (e.g., KTransformers, llama.cpp) is not modeled — resolver does not
  consider RAM as extended VRAM when selecting engines

### 3.2 Demand — Workload Matching

#### D1: `aima deploy <model>` single-command deploy — MVP, P0
- **Status: COMPLETE**
- Full pipeline: HAL detect → Resolve config → Runtime deploy (K3S/Docker/Native)
- `internal/runtime/runtime.go`: `Runtime` interface with `Deploy/Delete/Status/List/Logs`
- Three runtime implementations: K3S (`k3s.go`), Docker (`docker.go`), Native (`native.go`)

#### D2: Multi-model parallel deployment — MVP, P0
- **Status: COMPLETE**
- Multiple deployments tracked independently via deploy list
- Partition strategies in `catalog/partitions/` (3 YAML files)
- HAMi GPU partitioning support for K3S runtime

#### D3: Model format → engine auto-selection (with amplifier evaluation) — v1.0, P1
- **Status: PARTIAL**
- `InferEngineType()` (`internal/knowledge/resolver.go`) selects engine based on model format
  compatibility (`metadata.supported_formats`) and VRAM constraints (`vram_min_mib`)
- Format-to-engine mapping: `Catalog.FormatToEngine()` dynamically reads from YAML
- **Missing:** No amplifier evaluation — engine selection does not consider performance gain
  (FlashAttention, PagedAttention) or resource boundary expansion (CPU offload)

#### D4: App dependency declaration → auto-provision inference services — v1.0, P1
- **Status: MISSING**
- No "App" concept in code — no dependency declaration schema, no auto-provisioning
- PRD defines App as "inference consumer with dependency + resource budget + time constraints"
- No `inference_needs` or `resource_budget` schema implemented

#### D5: Time constraint awareness (startup time matching) — v1.0, P2
- **Status: MISSING**
- Engine YAML defines `cold_start_s` ranges (loaded into SQLite `engine_assets.cold_start_s_min/max`)
- Model YAML defines `startup_time_s` / `cold_start_time_s` per variant
- **Neither field is read by the resolver or used in deployment decisions**
- No `max_switch_time` constraint in deployment pipeline

### 3.3 Knowledge System

#### K1: Embedded recipes covering mainstream HW×Model combos — MVP, P0
- **Status: COMPLETE**
- 18 engine YAML files in `catalog/engines/` covering vLLM, llama.cpp, SGLang, TTS, ASR
  across NVIDIA (Ada/Blackwell), AMD ROCm, Hygon DCU, MUSA, Ascend, Iluvatar Spark
- 13 model YAML files in `catalog/models/` (Qwen3 family, GLM)
- 8 hardware profile YAML files in `catalog/hardware/`

#### K2: Hardware fingerprint matching → auto-select best recipe — MVP, P0
- **Status: COMPLETE**
- `findModelVariant()` matches by `gpu_arch` with VRAM and unified memory filtering
- `findEngine()` matches engine by `gpu_arch` compatibility
- `findContainerAccess()` matches vendor-specific container config from hardware profiles

#### K3: 5-layer config resolution (L0→L2c) with progressive override — MVP, P0
- **Status: COMPLETE (L0-L2b), PARTIAL (L2c)**
- `Catalog.Resolve()` (`internal/knowledge/resolver.go:67`) merges:
  L0 (engine defaults) → L2a (engine asset config) → L2b (model variant config) → L1 (user overrides)
- `ResolvedConfig.Provenance` tracks origin of each config value
- **L2c (tuning optimal):** DB schema exists for configurations + benchmarks, but no automatic
  promotion of benchmark-optimal configs into the resolve chain

#### K4: Recipes include performance data (tokens/s, startup time, power) — v1.0, P1
- **Status: PARTIAL**
- Engine YAML: `cold_start_s` ranges are loaded into SQLite
- Model YAML: `startup_time_s`, `cold_start_time_s` are defined but **not parsed by loader**
- Hardware YAML: `tdp_watts`, `power_modes` are loaded into SQLite
- **Gap:** These fields sit in SQLite tables but are never surfaced in `ResolvedConfig` or
  used by the deployment pipeline. The MCP tool `knowledge.search` can filter by `power_watts_max`
  (query.go:157) but this is post-hoc analysis, not pre-deployment guidance.

#### K5: Auto-feedback from tuning to knowledge base — v1.0, P1
- **Status: PARTIAL**
- `benchmark.record` MCP tool saves results to SQLite (`benchmark_results` table)
- `knowledge.promote` MCP tool manually promotes a configuration status
- **Missing:** No automatic cycle: benchmark → compare → promote best → inject into L2c resolve chain.
  Promotion is manual via MCP tool call.

#### K6: Central knowledge sync (push/pull) — v1.0, P1

#### K7: Offline knowledge export/import — v1.0, P1

#### K8: Knowledge query engine — multi-dimensional MCP tools — v1.0, P1

- `internal/knowledge/query.go`: `Store.Compare()` — side-by-side config comparison

- `internal/knowledge/query.go`: `Store.Lineage()` — `derived_from` chain traversal
- `internal/knowledge/query.go`: `Store.Aggregate()` — statistical summaries

#### K9: Central knowledge aggregation service (PostgreSQL + REST) — v1.0, P2

### 3.4 Control / Solver — Intelligent Scheduling

#### A1: Agent passive mode (user chat/MCP trigger) — MVP, P0

- **Status: MISSING**
- No `Patrol()`, `HealthMonitor()`, or scheduled task infrastructure

#### A3: Auto-tuning (parameter search + benchmark + apply optimal) — v1.0, P1
- **Status: MISSING**

  benchmarks each, compares results, and applies the winner

#### A4: Self-healing (detect → diagnose → recover) — v1.0, P1
- **Status: MISSING**
- No failure detection beyond K3S Pod status / Docker container status
- No diagnosis logic (e.g., OOM → reduce gpu_memory_utilization → redeploy)

#### A5: Engine switch cost evaluation (performance gain vs switch time) — v1.0, P2
- **Status: MISSING**
- `cold_start_s` data exists in YAML/SQLite but is never compared against performance benchmarks
- No cost-benefit analysis: "switching from llamacpp to vllm gains X tok/s but costs Y seconds downtime"
- Requires both time-constraint awareness (D5) and benchmark data comparison

## 4. YAML-Defined but Code-Unused Fields

These fields exist in catalog YAML, are parsed into Go structs, and loaded into SQLite,
but are **never consumed by the resolver, deployment pipeline, or constraint evaluation:**

| YAML Field | File(s) | SQLite Column | Go Struct Field | Used For |
|------------|---------|---------------|-----------------|----------|
| `cold_start_s` | `catalog/engines/*.yaml` (all 18 files) | `engine_assets.cold_start_s_min/max` | `EnginePerformance.ColdStartS` | Only stored; not used in engine selection or time constraint eval |
| `startup_time_s` | `catalog/models/qwen3.5-35b-a3b.yaml`, `qwen3-8b.yaml` | **Not loaded** | **Not parsed** | Defined in YAML but no Go struct field exists |
| `cold_start_time_s` | `catalog/models/qwen3.5-35b-a3b.yaml` | **Not loaded** | **Not parsed** | Defined in YAML but no Go struct field exists |
| `constraints.power_modes` | `catalog/hardware/*.yaml` | `hardware_profiles.power_modes` | `HardwareConstraints.PowerModes` | Stored as JSON; never queried or used for power constraint |
| `constraints.tdp_watts` | `catalog/hardware/*.yaml` | `hardware_profiles.tdp_watts` | `HardwareConstraints.TDPWatts` | Stored; never used for power budget enforcement |
| `hardware.gpu.compute_units` | `catalog/hardware/*.yaml` | Not stored | `GPUSpec.ComputeUnits` | Parsed but not stored in SQLite or used anywhere |
| `hardware.gpu.compute_id` | `catalog/hardware/*.yaml` | `hardware_profiles.gpu_compute_id` | `GPUSpec.ComputeID` | Stored; not used in matching or selection |
| `hardware.ram.bandwidth_gbps` | `catalog/hardware/*.yaml` | Not stored | `RAMSpec.BandwidthGbps` | Parsed but not stored or used |
| `partition.cpu_tools` | `catalog/hardware/*.yaml` | Not stored | `HardwarePartition.CPUTools` | Parsed but not used (no ResourceSlot backend) |

### Impact

These fields represent **pre-invested knowledge** that the PRD's optimization model depends on:
- `cold_start_s` + `startup_time_s` → needed for time constraint evaluation (D5, A5)
- `tdp_watts` + `power_modes` → needed for energy constraint evaluation (S3 full, F4 full)
- `compute_units` + `bandwidth_gbps` → needed for resource budget estimation (S4)
- `cpu_tools` → needed for ResourceSlot pluggable backends (S5)

Connecting these fields to the resolver and deployment pipeline is the highest-leverage work
for closing v1.0 gaps, as the data is already present.

## 5. Implementation Roadmap

### Tier 1 — High-Impact, Go Code Only (no new services)

These can be implemented entirely within the existing Go binary by connecting existing
data (YAML fields, SQLite columns) to existing logic (resolver, deploy pipeline).

| Gap | PRD IDs | Approach | Key Files |
|-----|---------|----------|-----------|
| Time constraint in resolver | D5, A5 | Add `cold_start_s` / `startup_time_s` to `ResolvedConfig`; filter engines by `max_switch_time` | `resolver.go`, `loader.go` |
| Power budget enforcement | S3 full, F4 | Compare `tdp_watts` against deployment power estimate; warn/block on exceed | `resolver.go`, `CheckFit()` |
| L2c auto-promote | K5 | After benchmark, if result beats current best for same HW×Engine×Model, auto-promote config | `cmd/aima/main.go` (benchmark wiring) |
| Resource budget estimation | S4 | Estimate VRAM/RAM cost from model variant metadata before deploy; expose as dry-run field | `resolver.go`, `podgen.go` |
| Power monitoring endpoint | F4 | Add `/api/v1/power` aggregating HAL power data over time | `internal/proxy/` |
| Model variant startup_time_s parsing | K4 | Add `StartupTimeS` / `ColdStartTimeS` to `ModelVariant` struct in `loader.go` | `loader.go` |

### Tier 2 — Agent Orchestration (Go code + LLM integration)

These require the Go Agent (L3a) or an external Agent to execute multi-step workflows.
The Go infrastructure (MCP tools) already exists; what's missing is the orchestration logic.

| Gap | PRD IDs | Approach | Key Files |
|-----|---------|----------|-----------|
| Proactive patrol | A2 | Add `PatrolLoop()` with configurable ticker: call `device.metrics` → check thresholds → alert | `internal/agent/` (new file) |
| Auto-tuning | A3 | Agent loop: enumerate config variations → `benchmark.run` each → `knowledge.promote` best | Agent prompt engineering or `internal/agent/tuner.go` |
| Self-healing | A4 | In patrol loop: detect failed deploy → diagnose (OOM? crash?) → auto-remediate or alert | `internal/agent/` |
| Engine switch evaluation | A5 | Compare `cold_start_s` cost vs benchmark throughput gain; surface as MCP tool recommendation | `internal/knowledge/` (new query) |
| Knowledge effectiveness validation | F5 | Post-deploy: run benchmark → compare vs recipe prediction → flag divergence | Agent workflow |
| Open questions resolution | I6 | Agent reads `open_questions` from stack YAML → tests on device → updates knowledge | Agent workflow |

### Tier 3 — New Services / Major Architecture (beyond edge binary)

| Gap | PRD IDs | Approach | Notes |
|-----|---------|----------|-------|
| Central knowledge server | K6, K9 | Separate Go service: PostgreSQL + REST API (`/api/v1/ingest`, `/api/v1/query`, `/api/v1/sync`) | New project or module |
| Knowledge push/pull client | K6 | Add `knowledge.SyncClient` in edge binary, call central REST endpoints | `internal/knowledge/sync.go` |
| Tunnel remote access | — | Integrate Tailscale/WireGuard or build custom tunnel; expose MCP over WAN | Architecture decision needed |
| TUI dashboard | F6 | Bubble Tea or similar Go TUI framework; lower priority given Web UI exists | `internal/tui/` |
